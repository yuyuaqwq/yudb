
各个模块都经历了非常精心的设计及对难点的攻克，如页面管理器
    基于三级页表
减少io次数：
    最坏只需要2次io即可管理4TB空间，且由于页面的缓存，绝大多数分配释放都不会产生io。
低页面碎片：
    多级页表中又基于伙伴系统管理下级页表，分配以2为底的幂为单位的连续页，基于任何数都可以被多个以2为底的幂组成的性质来管理过长的value，即不会有太多随机io，也保证了页面利用率，及更低的页面碎片。
低管理开销：
    每一页管理的开销压缩到≈4字节，且管理所占的空间是动态扩展的。



空闲表：
    页面通过多级空闲表来管理，当前分2级
    类型分配PageTable和DirTable
    
    f0是最高的一级，常驻内存
    f1存放与其他页面一致通过页面缓存加载
    
    数据库布局如下：
    |meta0 + meta1 + f0table_0 + f0table_1 + f1table_0 + f1table_1 + data|f1table_0 + f1table_1 + data|...|f1table_0 + f1table_1 + data|

表结构：
    f0:
        int16_t list_first[4];
        f0_entry entry[1021];
        f0的entry只能存1021项，因此max_count为1021，f0_entry[0]表示第一个f0
    
    f1:
        int16_t list_first[4];
        f1_entry entry[1022];
        f1的entry只能存1022项，但是前2项正好对应了不可分配的f1_table，因此max_count为1024，f1_entry[0]表示第三个f1
        
    对于f0.entry[0]而言，f1.entry初始时，前6页都是不可用的(除开默认前2页，还需除开4页)


修改：
    f0常驻内存中
        若是原地更新模式，每个事务都会切换f0_0/f0_1
        若是wal模式，持续使用同一个f0
    f1通过常规的缓存加载
        f0_entry在数据库创建时指向f1_0为写，f1_1为读，即f1_1是已持久化的版本
        原地更新模式：
            当f1_1读入后，写回就是写回到f1_0，再次读取也是f1_0(当前写事务而言是更新的版本，此时read==write)
            提交事务时从f0_entry交换其读写，f1_1为写，f1_0为读
        wal模式：
            当f1_1读入后，写回就是写回到f1_0，再次读取也是f1_0(当前写事务而言是更新的版本，此时read==write)
            提交事务什么都不做，下次再开启事务时，f1_1依旧是最后已持久化的版本，读写依旧是f1_0(read==write)
            关闭数据库时，才会从f0_entry交换其读写，f1_1为写，f1_0为读


伙伴系统管理：
    f0_dir_table:
        uint8 buddy_entry[1024];
        int16_t list_first[4];  // Alloc、Full、Pending、保留
        f0_entry entry[1021];  // 末3页无法管理，因为list_first需要占8字节
       
    f0_entry(3byte):
        uint8_t f1_max_free_log : 4;        // f1最大连续空闲计数，用于快速判断f1是否足够分配，避免可能的io
        uint8_t : 4;
        
        uint16_t read_select : 1;       // 当前f0_entry读取f1_table时的索引，初始为1
        uint16_t write_select : 1;       // 当前f0_entry落盘f1_table时的索引，初始为0
        int16_t list_next : 14;     // 静态链表

    f0最大空闲计数是1021
    
    f1_table:
        uint8 buddy_entry[1024];
        int16_t list_first[2];
        f1_entry entry[1022];  // 因为前2页正好会被f1_table占用，正好不需要管理，提前在buddy占位即可
        
    f1_entry(3byte):
        uint8_t reserved;
        uint16_t list_next;     // 静态链表

    初始化前，最大空闲计数是1024，初始化后，最大空闲计数应该是1022
    
    分配小于等于512的页面时，从f1的伙伴系统分配，否则从f0分配，1个f0_entry表示1024个页面


解决内部碎片：
	基于任意数都可以被2的幂组成的性质，大空间通过多次页面分配，并使用一个目录管理(这个目录可以在当前页面中分配一块内存，在element中指向)
	如800页，则分成512+256+32




页面分配释放：
    被写事务释放后，不能被直接分配的页面为pending页面，可能有其他读事务还需要访问
    此时在空闲表中以pending的状态存在，等到下一个写事务开始才可能会真正释放
    但是涉及到页面管理器的频繁底层调用，性能较差
    如果将其临时保留到内存中，此时空闲表中还是已被分配的状态
    原地更新模式：
        事务执行时，空闲表直接为pending状态，分配释放直接从空闲表中进行
        暂不考虑优化，写入瓶颈在随机io
    wal模式：
        创建两个页号池，pending和free
        pending池的pending页面有两种状态：已持久化pending和未持久化pending
            实际实现在事务管理器中记录了未持久化pending
        pending操作不会实际调用空闲表pending，而是放到pending池中(空闲表中是分配状态)
        如果读事务嘎了，新写事务就把对应的pending池的页面(只能是未持久化pending)拿出来放到free池中
        页面分配优先从free池，没有再调用空闲表接口
        由于已持久化pending在空闲表中是分配状态，所以不会被分配
        释放也是直接放到free池
        
        持久化版本切换时，再将pending池的pending状态落盘，以及free池的空闲页面状态落盘
